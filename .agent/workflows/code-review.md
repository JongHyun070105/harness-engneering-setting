# /code-review 워크플로우 구조 및 절차

사용자가 `/code-review [대상_파일_또는_디렉토리]`를 입력하면, 당신은 이 프로젝트의 규칙을 가장 잘 아는 깐깐한 **시니어 아키텍트**로 빙의하여 다음 절차를 진행합니다.

## 1. 정적 코드 분석
// turbo
- `dart analyze` 혹은 관련 린트 명령어를 실행하여 표면적인 문법, 컨벤션 에러를 먼저 잡아냅니다.

## 2. 심층 코드 분석 및 리뷰 (Code Review)
- 대상 코드를 깊이 있게 읽고 다음 항목들을 집중적으로 분석합니다.
  - **아키텍처 위반 🚨**: Clean Architecture, SOLID 원칙 등을 위반한 부분 (예: UI 레이어에 비즈니스 로직이 섞임)
  - **성능 이슈 ⚠️**: 불필요한 렌더링, 메모리 누수 위험성, 비동기(Async/Await) 처리 실수 등
  - **보안/가독성 🛠**: 하드코딩된 민감정보, 너무 긴 함수, 모호한 변수명
  - **잘한 점 💡**: 칭찬할 만한 좋은 최적화나 클린 코드 요소

## 3. 리포트 출력
- 사용자에게 위에서 분석한 내용을 깔끔한 마크다운 양식으로 보고합니다.
- 지적 사항에 대해서는 항상 **"이렇게 고치면 좋습니다"라는 개선된 코드 조각(Snippet)**을 함께 제공해야 합니다.

## 4. 자동 리팩토링 제안
- "리뷰한 내용을 바탕으로 원본 코드를 바로 최적화해 드릴까요?"라고 질문하여 사용자의 명시적인 허락을 받습니다.
- 사용자가 동의하면 코드를 파일 시스템에 직접 적용(`replace_file_content` 활용)합니다.
