# Harness 12 Principles — Flutter/Antigravity 맞춤

---

## P1. Agent Entry Point

### 정의
에이전트가 프로젝트에 처음 진입했을 때, 어디서부터 시작해야 하는지를 안내하는 진입점이 있는가?

### 판단 기준
- [ ] `.agent/rules/global.md`가 존재하고 프로젝트 개요를 포함하는가?
- [ ] 빌드/테스트/린트 명령어가 실행 가능한 형태로 기술되어 있는가?
- [ ] 에이전트가 반드시 지켜야 할 핵심 규칙(금지 사항)이 명시되어 있는가?
- [ ] 하위 프로젝트에 `.agent/rules/project.md`가 존재하는가?
- [ ] `docs/gotchas.md` 또는 `rules/gotchas.md`로 실패 기록이 참조되는가?

### 점수 기준
| 점수 | 기준 |
|---|---|
| 0 | 진입점 파일 없음 |
| 3 | README.md만 존재, 에이전트 전용 안내 없음 |
| 5 | `global.md` 존재하나 불완전 |
| 8 | 진입점 완비, 하위 프로젝트 안내 부족 |
| 10 | 루트 + 프로젝트별 진입점 완비, 정기 갱신 |

## P2. Map, Not Manual

### 정의
문서가 백과사전이 아닌 탐색 가능한 지도(목차) 역할을 하는가?

### 판단 기준
- [ ] `global.md`가 100줄 이하로 간결한가?
- [ ] 상세 규칙이 별도 문서(`docs/conventions.md` 등)로 분리되어 있는가?
- [ ] 문서 간 상호 참조(링크)가 있는가?
- [ ] 스킬 파일이 진입점(SKILL.md) → 참조 문서(references/) 구조인가?

### 점수 기준
| 점수 | 기준 |
|---|---|
| 0 | 문서 없음 또는 수백 줄 모놀리식 |
| 3 | 문서 있으나 분리 안 됨 |
| 5 | 일부 분리, 상호 참조 부족 |
| 8 | Map 구조, 90% 이상 상호 참조 |
| 10 | 완전한 계층 구조 + 모든 문서 상호 연결 |

## P3. Invariant Enforcement

### 정의
도구(lint, type check, test, 아키텍처 검증)가 에이전트의 실수를 자동으로 잡는가?

### 판단 기준
- [ ] `analysis_options.yaml`에 린트 규칙이 설정되어 있는가?
- [ ] `dart analyze`가 체크포인트에서 실행되는가?
- [ ] `dart format`이 자동 적용되는가?
- [ ] 아키텍처 검증 스크립트(`check_architecture.sh` 등)가 존재하는가?
- [ ] 테스트 실행이 프로토콜에 포함되어 있는가?

### 점수 기준
| 점수 | 기준 |
|---|---|
| 0 | 자동화 도구 없음 |
| 3 | lint만 설정 |
| 5 | lint + format 자동화 |
| 8 | lint + format + test + 아키텍처 검증 |
| 10 | 위 모두 + pre-commit hook + CI 강제 |

---

## P4. Convention Over Configuration

### 정의
암묵적 규칙이 아닌 명시적 컨벤션이 문서로 존재하는가?

### 판단 기준
- [ ] 코딩 컨벤션 문서(`docs/conventions.md`)가 존재하는가?
- [ ] 네이밍 규칙(파일, 클래스, 변수)이 명시되어 있는가?
- [ ] Git 커밋 컨벤션이 정의되어 있는가?
- [ ] Good/Bad 예시가 포함되어 있는가?

### 점수 기준
| 점수 | 기준 |
|---|---|
| 0 | 컨벤션 문서 없음 |
| 3 | 간단한 스타일 가이드만 |
| 5 | 컨벤션 문서 존재, 예시 부족 |
| 8 | 상세 컨벤션 + Good/Bad 예시 |
| 10 | 완전한 컨벤션 + 도구로 강제 + 정기 갱신 |

## P5. Progressive Disclosure

### 정의
에이전트가 필요한 정보를 필요한 시점에 점진적으로 접근할 수 있는가?

### 판단 기준
- [ ] 정보가 최소 2단계 계층 (요약 → 상세)으로 구성되어 있는가?
- [ ] 진입점에서 상세 문서로의 경로가 명확한가?
- [ ] 스킬이 필요할 때만 로드되는 구조인가?

### 점수 기준
| 점수 | 기준 |
|---|---|
| 0 | 모든 정보가 한 곳에 뭉쳐있음 |
| 5 | 2단계 분리 |
| 8 | 3단계 분리 + 참조 링크 |
| 10 | 3단계 + 각 계층 완비 + 적시 로딩 |

---

## P6. Layered Architecture

### 정의
의존성 방향이 명확하고 도구로 강제되는가?

### 판단 기준
- [ ] 레이어 분리가 되어 있는가? (presentation/domain/data)
- [ ] 의존성 방향이 일관적인가? (순환 의존성 없음)
- [ ] Import 규칙이 도구(`check_architecture.sh`)로 강제되는가?
- [ ] 모듈 경계가 명확한가?

### 점수 기준
| 점수 | 기준 |
|---|---|
| 0 | 레이어 분리 없음 |
| 3 | 디렉토리 분리만, 규칙 없음 |
| 5 | 분리 + 문서 규칙 |
| 8 | 분리 + 스크립트 강제 |
| 10 | 분리 + 스크립트 + CI + 순환 의존성 0 |

## P7. Garbage Collection

### 정의
낡은 코드/문서가 정기적으로 정리되고 있는가?

### 판단 기준
- [ ] dead code 탐지 수단이 있는가?
- [ ] 문서가 코드 변경을 반영하고 있는가? (Doc Drift 없음)
- [ ] 불필요한 의존성이 정리되어 있는가?

### 점수 기준
| 점수 | 기준 |
|---|---|
| 0 | 정리 미시행 |
| 5 | 수동 정리 |
| 8 | 반자동 정리 + 정기 점검 |
| 10 | 자동 탐지 + 정기 정리 + drift < 2% |

---

## P8. Observability

### 정의
에이전트가 작업 결과를 스스로 검증할 수 있는가?

### 판단 기준
- [ ] 체크포인트(dart analyze + test)가 프로토콜에 포함되어 있는가?
- [ ] 피드백 루프 소요 시간이 3분 이내인가?
- [ ] SDD 명세 대조 검증이 프로토콜에 포함되어 있는가?
- [ ] gotchas 기록 루프가 있는가?

### 점수 기준
| 점수 | 기준 |
|---|---|
| 0 | 자체 검증 수단 없음 |
| 5 | 수동 검증만 |
| 8 | 자동 체크포인트 + 명세 대조 |
| 10 | 위 모두 + 피드백 루프 < 1분 + gotchas 자동 참조 |

---

## P9. Knowledge in Repo

### 정의
개발에 필요한 지식이 사람의 머릿속이 아닌 레포에 있는가?

### 판단 기준
- [ ] 아키텍처 결정이 레포 내에 기록되어 있는가?
- [ ] gotchas(실패 기록)가 레포에 있는가?
- [ ] 온보딩에 필요한 모든 정보가 레포에 있는가?
- [ ] 외부 문서 의존도가 20% 미만인가?

### 점수 기준
| 점수 | 기준 |
|---|---|
| 0 | 구전 지식 의존 |
| 5 | 일부 문서화 |
| 8 | 대부분 레포 내 + 외부 의존 < 20% |
| 10 | 레포만으로 즉시 작업 가능 + 정기 갱신 |

## P10. Reproducibility

### 정의
같은 입력에 같은 결과가 보장되는가?

### 판단 기준
- [ ] `pubspec.lock`이 버전 관리되는가?
- [ ] Flutter/Dart SDK 버전이 고정되어 있는가?
- [ ] 환경 변수 설정이 문서화되어 있는가?
- [ ] 빌드 명령어가 복사-붙여넣기로 실행 가능한가?

### 점수 기준
| 점수 | 기준 |
|---|---|
| 0 | "내 컴에서는 되는데" 빈발 |
| 5 | lock 파일 + SDK 버전 고정 |
| 8 | 위 + 환경 문서화 + Docker 또는 FVM |
| 10 | 완전 재현 가능 + CI에서 검증 |

---

## P11. Modularity

### 정의
변경의 영향 범위가 예측 가능한가?

### 판단 기준
- [ ] feature 간 직접 import가 없는가?
- [ ] 공통 코드가 `core/`로 중앙화되어 있는가?
- [ ] 1개 변경 시 영향받는 파일이 3개 이하인가?

### 점수 기준
| 점수 | 기준 |
|---|---|
| 0 | 스파게티 코드 |
| 5 | 디렉토리 분리, 일부 순환 의존 |
| 8 | 명확한 모듈 경계, 영향 범위 예측 가능 |
| 10 | 완전 격리 + 영향 파일 평균 1개 + 도구 강제 |

---

## P12. Self-Documentation

### 정의
코드 자체가 의도를 설명하는가?

### 판단 기준
- [ ] 변수명/함수명만으로 의도를 파악할 수 있는가?
- [ ] 매직 넘버가 상수로 추출되어 있는가?
- [ ] 주석이 "what" 대신 "why"를 설명하는가?
- [ ] 파일명과 디렉토리 구조만으로 코드 위치를 예측할 수 있는가?

### 점수 기준
| 점수 | 기준 |
|---|---|
| 0 | 네이밍 불규칙, 매직 넘버 산재 |
| 5 | 일부 개선, 불일관적 |
| 8 | 일관된 네이밍 + 상수 추출 + why 주석 |
| 10 | 외부 문서 없이 코드만으로 완전 이해 가능 |

## 우선순위 (ROI 기준)

1. **P3 Invariant Enforcement** — 자동 강제가 에이전트 오류율을 극적으로 감소
2. **P1 Agent Entry Point** — 진입점 하나로 초기 탐색 시간 90% 단축
3. **P8 Observability** — 자체 검증 → 결과 품질 보장
4. **P4 Convention** — 일관된 패턴 → 패턴 매칭 극대화
5. **P9 Knowledge in Repo** — 레포에 지식 → 에이전트 접근 가능
6. **P10 Reproducibility** — 동일 결과 → 시행착오 감소
7. **P11 Modularity** — 변경 범위 예측 → 안전한 수정
8. **P6 Layered Architecture** — 의존성 방향 → 부작용 방지
9. **P12 Self-Documentation** — 코드 가독성 → 이해도 직결
10. **P2 Map, Not Manual** — 탐색 효율 개선
11. **P5 Progressive Disclosure** — 정보 과부하 방지
12. **P7 Garbage Collection** — 거짓 신호 제거
